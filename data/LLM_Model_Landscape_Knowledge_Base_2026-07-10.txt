This knowledge base is a dated technical landscape for building and evaluating retrieval-augmented generation systems. It covers 66 model or model-family cards across 15 providers, plus retrieval, document parsing, safety, deployment, cost, evaluation, governance, and lifecycle guidance. The research cut-off is 2026-07-10.

The model market changes quickly. Names, aliases, context windows, supported regions, preview status, quotas, prices, and retirement dates can change independently. Every numerical statement should therefore be treated as a snapshot, not a permanent contract. Before procurement or production release, reopen the official URLs in the source registry and verify the exact endpoint, region, billing unit, service tier, currency, tax treatment, and license.

Research method:
- Prefer first-party provider documentation, official model cards, pricing pages, cloud catalogs, and lifecycle notices.
- Separate provider claims from implementation advice. Provider claims are labeled as such; failure analysis and selection guidance are analytical synthesis.
- Avoid ranking models with one universal score. Real performance depends on the prompt, language, domain, retrieval corpus, tools, safety rules, context length, and infrastructure.
- Treat “open-weight” as a deployment category, not a guarantee of an unrestricted open-source license.
- Include non-generative components—embeddings, rerankers, OCR/document parsers, moderation, and realtime speech—because a reliable RAG system is a pipeline, not only a chat model.

How to use the three files:
- The PDF is the human-readable reference and training handout.
- The DOCX is editable and suitable for adding internal evaluation results, procurement notes, and organization-specific policies.
- The TXT file is deliberately chunked with metadata delimiters for ingestion into a vector database or hybrid search index.

doc_id: guide-02-executive-selection-guide
title: Executive selection guide
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, executive-selection-guide
source_ids: 
---CONTENT---
There is no universally best model. Select a portfolio by workload.

Use a frontier or strong reasoning model for difficult synthesis, coding, planning, ambiguous questions, and high-value agent decisions. Route simple tasks away from it. Use a balanced model as the default generator for enterprise chat and RAG. Use a fast or small model for query rewriting, classification, extraction, routing, guardrails, and high-volume FAQs. Use an open-weight model when data locality, customization, predictable sustained-volume economics, or sovereign deployment justifies operating the infrastructure.

Use a search-grounded model for current public information, but keep public web evidence separate from private-corpus evidence. Use a vision model when layout, images, screenshots, charts, or scanned pages carry meaning. Use OCR or a document parser to create stable text, table, and provenance records. Use embeddings for first-stage semantic retrieval, lexical search for exact identifiers, and a reranker to improve evidence ordering. Use a safety model as one defense layer, never as authorization.

A strong production pattern is a cascade:
1. Authenticate the user and derive access filters.
2. Classify the request for domain, risk, freshness, modality, and complexity.
3. Retrieve with hybrid search and mandatory metadata filters.
4. Rerank and diversify evidence.
5. Route easy questions to a cheap model and difficult questions to a stronger model.
6. Validate citations, schema, policy, and tool parameters.
7. Escalate low-confidence or high-risk results to a human.
8. Log the model version, prompt version, retrieval set, costs, latency, and outcome.

doc_id: guide-03-model-taxonomy-and-terminology
title: Model taxonomy and terminology
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, model-taxonomy-and-terminology
source_ids: 
---CONTENT---
Frontier model: a provider’s strongest general model, usually with higher cost and latency.
Balanced model: a production-oriented compromise among quality, speed, and price.
Fast model: optimized for throughput and response time; often suitable for utility stages.
Reasoning model: allocates extra inference to multi-step problems. Reasoning is not evidence and is not guaranteed correct.
Small language model: a compact model suitable for low-cost, private, or narrow tasks.
Open-weight model: downloadable parameters are available under a specified license. This does not automatically mean the complete training data, training code, or unrestricted commercial rights are open.
Multimodal model: accepts or produces more than one modality, such as text, image, audio, or video.
Context window: the maximum combined input—and sometimes output—token budget accepted by an endpoint. Advertised capacity does not guarantee uniform recall or economical use.
Output limit: maximum generated tokens; it can be far smaller than the context window.
Embedding model: produces vectors used for similarity search; it does not generate a natural-language answer.
Reranker: scores query-document pairs after broad retrieval.
OCR/document parser: extracts text, tables, reading order, coordinates, and document structure.
Search-grounded model: invokes public search and cites retrieved web pages.
Prompt caching: discounted or accelerated reuse of a matching prompt prefix, subject to provider-specific rules.
Batch API: asynchronous or delayed processing offered at lower prices by some providers.
Tool/function calling: structured requests from a model to application-defined functions. It is not permission to execute them.
Structured output: generation constrained to a schema. Validation is still required.

doc_id: guide-04-pricing-and-total-cost-methodology
title: Pricing and total-cost methodology
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, pricing-and-total-cost-methodology
source_ids: 
---CONTENT---
Token prices are not directly comparable until the billing assumptions match. A practical estimate is:

monthly model cost =
(input tokens × input rate)
+ (cached input tokens × cached-input rate)
+ (output tokens × output rate)
+ reasoning-token charges
+ search/grounding/request fees
+ image/audio/video units
+ tool or hosted-compute charges
+ batch and service-tier adjustments.

Then add retrieval, vector database, reranking, object storage, egress, observability, evaluation, security, and engineering operations.

Example comparison method:
- Build a fixed workload of at least several hundred representative requests.
- Record actual tokenizer counts rather than estimating from character count.
- Measure average and percentile output length; output is often much more expensive than input.
- Include retries, timeouts, safety re-prompts, agent loops, and failed tool calls.
- Compare quality at an agreed service-level objective, not at equal token count.
- Model cached and uncached traffic separately.
- For web-grounded APIs, include request/search/citation fees.
- For realtime audio, use the provider’s audio billing units and session-duration assumptions.
- For self-hosting, calculate cost per successful request at realistic utilization, including idle capacity and redundancy.

Do not put a provider price directly into application logic. Keep a dated price catalog, alert on changes, and attach the price version to cost reports.

doc_id: guide-05-reference-rag-architecture
title: Reference RAG architecture
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, reference-rag-architecture
source_ids: 
---CONTENT---
A dependable RAG pipeline has two paths: ingestion and query serving.

Ingestion:
1. Register the source with owner, classification, jurisdiction, retention, access policy, checksum, and effective date.
2. Parse the native file while preserving page numbers, headings, tables, figures, code blocks, and coordinates.
3. Apply OCR only where needed and retain confidence scores and original page images.
4. Normalize carefully; do not destroy identifiers, units, legal numbering, or table relationships.
5. Chunk by structure and meaning. Keep parent-child links and overlap only where it improves continuity.
6. Add metadata such as source_id, section_path, page, effective_from, effective_to, language, tenant, and ACL.
7. Create lexical and dense indexes. Store the embedding model and preprocessing version.
8. Run ingestion quality checks and sample visual comparisons.
9. Publish an immutable corpus version; use incremental updates and tombstones for deletions.

Query serving:
1. Authenticate and calculate the user’s permissions.
2. Detect language, intent, domain, time sensitivity, risk, and required modalities.
3. Rewrite or decompose the query without losing named entities, dates, identifiers, and constraints.
4. Retrieve with dense, lexical, structured, graph, or SQL methods as appropriate.
5. Apply authorization filters before candidates reach the model.
6. Rerank and diversify; prefer authoritative and current sources.
7. Pack a bounded context with stable chunk IDs and clear delimiters.
8. Generate an answer under a policy that forbids unsupported claims.
9. Validate citations, numbers, JSON/schema, sensitive data, and tool parameters.
10. Return evidence links and log the full trace for evaluation.

The generator cannot repair evidence that was never retrieved. Measure corpus coverage, parser accuracy, retrieval recall, reranker quality, and answer grounding separately.

doc_id: guide-06-chunking-and-metadata-design
title: Chunking and metadata design
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, chunking-and-metadata-design
source_ids: 
---CONTENT---
Chunking should follow document structure rather than a universal character count.

Narrative policies and manuals: split on headings, then paragraphs, with modest overlap when references cross boundaries.
Tables: retain headers, units, row/column relationships, and a link to the original table or page image. Consider one chunk per logical row group plus a table summary.
Code: chunk by file, class, function, or symbol; preserve repository path, commit, language, imports, and call relationships.
Contracts and regulations: preserve section numbers, definitions, amendments, effective dates, and cross-references.
Slides: keep slide number, title, speaker notes, and visual description.
Scans: retain OCR confidence and bounding boxes.
Conversations: separate participant, timestamp, thread, and message-level permissions.

Recommended metadata:
doc_id, chunk_id, parent_id, source_uri, source_type, title, provider, model_id, section_path, page_start, page_end, language, jurisdiction, effective_from, effective_to, created_at, updated_at, ingested_at, corpus_version, parser_name, parser_version, embedding_model, embedding_dimension, content_hash, tenant_id, owner, confidentiality, acl, pii_flags, tags, and source_authority.

Do not place the only copy of critical metadata inside the embedded text. Use filterable structured fields. Treat document-level and chunk-level metadata separately.

doc_id: guide-07-retrieval-reranking-and-context-packing
title: Retrieval, reranking, and context packing
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, retrieval-reranking-and-context-packing
source_ids: 
---CONTENT---
Hybrid retrieval is a strong default because semantic and lexical methods fail differently. Dense vectors find paraphrases and concepts. BM25 or another lexical method finds exact product codes, names, error messages, legal citations, and rare terms. Structured filters enforce tenant, language, date, document type, and authorization. Graph or SQL retrieval is better when relationships or exact aggregation matter.

Retrieve broadly enough to protect recall, then rerank a bounded set. Measure retrieval Recall@k, MRR, nDCG, and coverage on real queries. Reranking should not remove all diversity: multiple independent passages may be necessary to answer a question or reveal a contradiction.

Context packing rules:
- Put instructions outside and above retrieved content.
- Mark retrieved text as untrusted data, never as instructions.
- Keep stable chunk IDs next to every passage.
- Prefer fewer complete, authoritative passages to many partial duplicates.
- Preserve headings, dates, units, and table context.
- Order evidence deliberately; test primacy, recency, and lost-in-the-middle effects.
- Reserve space for output and tool calls.
- Do not exceed a model’s context window merely because the API accepts a large prompt.

doc_id: guide-08-failure-modes-and-mitigations
title: Failure modes and mitigations
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, failure-modes-and-mitigations
source_ids: 
---CONTENT---
Hallucination: the model states unsupported facts. Mitigate with evidence-only prompting, no-answer behavior, citation validation, and human review.
Retrieval miss: the right chunk is absent. Improve parsing, metadata, query decomposition, hybrid search, synonyms, and corpus coverage.
Retrieval noise: distractors dominate. Use reranking, deduplication, authority weighting, and smaller context.
Citation mismatch: a cited source does not support the sentence. Validate claim-to-passage entailment and show quoted evidence snippets where appropriate.
Prompt injection: a document tells the model to ignore policy or call a tool. Treat documents as untrusted, isolate instructions, restrict tools, and scan content.
Authorization leak: retrieval returns another user’s data. Enforce ACL filters in the retrieval layer; never rely on the model to hide it.
Stale answer: an older document outranks the effective version. Model effective dates and authority, remove superseded chunks, and display freshness.
Long-context degradation: evidence is accepted but ignored. Retrieve selectively, summarize hierarchically, and test evidence placement.
Multilingual drift: meaning or policy changes across languages. Evaluate per language and preserve original passages.
Numeric error: totals, dates, or units are misread. Use deterministic calculators, SQL, schema checks, and source-value comparison.
Tool error: the model generates invalid or dangerous arguments. Validate schemas and permissions, use idempotency, confirmations, and allowlists.
Format drift: output violates JSON or a template. Use structured outputs, parsers, retries, and deterministic post-validation.
OCR/layout error: text order or table relationships are corrupted. Preserve images and coordinates and route low-confidence cases for visual review.
Model update regression: an alias changes behavior. Pin versions where possible and gate every change through regression tests.

doc_id: guide-09-evaluation-framework
title: Evaluation framework
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, evaluation-framework
source_ids: 
---CONTENT---
Build an evaluation set from real user tasks, historical failures, domain-expert examples, and adversarial tests. Separate answerable and unanswerable questions. Label the required evidence and acceptable answer variants.

Retrieval metrics:
- Recall@k: whether required evidence appears in the candidate set.
- MRR and nDCG: ranking quality.
- ACL precision: whether unauthorized chunks are always excluded.
- freshness/authority accuracy: whether current and authoritative documents are preferred.

Generation metrics:
- correctness and completeness,
- faithfulness to evidence,
- citation precision and recall,
- no-answer precision and recall,
- instruction and schema compliance,
- tone and domain terminology,
- multilingual quality,
- tool-call validity,
- safety and privacy outcomes.

Operational metrics:
- end-to-end p50/p95/p99 latency,
- time to first token,
- successful requests per dollar,
- input, cached input, reasoning, and output tokens,
- retrieval and reranking latency,
- retry and timeout rate,
- human escalation rate,
- user correction and abandonment rate.

Use pairwise human review for meaningful model comparisons. Blind reviewers to provider names. Define materiality: a cheaper model is not better if it creates more escalations or incorrect business actions.

---CONTENT---
A RAG system expands the attack surface because it connects models, proprietary documents, identity systems, vector databases, external tools, and logs.

Core controls:
- identity-aware retrieval with deny-by-default filters,
- encryption in transit and at rest,
- tenant isolation and document-level ACLs,
- secrets management and short-lived credentials,
- data-loss prevention for prompts, outputs, and logs,
- explicit retention and deletion propagation into indexes,
- provider data-use and residency review,
- tool allowlists, parameter validation, and least privilege,
- prompt-injection and exfiltration testing,
- moderation and abuse monitoring,
- immutable audit records for sensitive actions,
- human approval for consequential decisions.

Do not log raw sensitive prompts by default. Redact or tokenize PII, credentials, health data, and financial identifiers. Keep a data map showing which components receive raw documents, extracted text, vectors, prompts, and generated outputs.

Governance records should include model and prompt versions, model card, intended use, prohibited use, evaluation results, risk owner, data sources, licenses, region, vendor terms, incident process, retirement plan, and approval status.

doc_id: guide-11-agents-and-tool-use
title: Agents and tool use
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, agents-and-tool-use
source_ids: 
---CONTENT---
An LLM agent is a policy-driven loop around a model, tools, memory, and state. The model proposes actions; the application decides whether they are valid and permitted.

Design principles:
- Prefer deterministic workflows when the steps are known.
- Give each tool a narrow schema, clear description, bounded output, and least-privilege credential.
- Separate read tools from write tools.
- Validate every argument outside the model.
- Require confirmation for irreversible, expensive, external, or sensitive actions.
- Use idempotency keys and transaction boundaries.
- Set maximum steps, time, token, and spend budgets.
- Detect repeated plans and stop loops.
- Keep retrieved text and tool output marked as untrusted.
- Recheck authorization at execution time.
- Summarize state rather than allowing an unbounded conversation transcript.
- Record action, evidence, model, tool result, and final disposition.

A stronger model may improve planning but does not replace workflow controls. Agent reliability is a systems-engineering property.

---CONTENT---
Multimodal systems should keep modality-specific evidence instead of converting everything into an untraceable paragraph.

For images and pages:
- preserve the original image, resolution, page number, crop coordinates, and OCR output;
- use a document parser for reading order and tables;
- use a vision model when visual relationships matter;
- test small text, rotated scans, charts, legends, and handwriting;
- display the source page for human verification.

For voice:
- store a timestamped transcript subject to consent and retention rules;
- confirm names, numbers, dates, addresses, and high-risk instructions;
- handle interruptions, silence, noise, accents, and crosstalk;
- measure first-audio latency and turn-end detection;
- retrieve textual evidence before speaking the answer;
- provide citations in a companion visual interface where possible.

For video:
- sample frames and align them with transcript timestamps;
- avoid assuming a model observed every frame;
- preserve temporal provenance for claims.

title: Deployment patterns and model routing
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, deployment-patterns-and-model-routing
source_ids: 
---CONTENT---
Managed API:
Best for rapid delivery, elastic scale, and access to frontier capability. Risks include variable price, vendor dependency, data-policy constraints, regional availability, quotas, and model lifecycle.

Cloud model catalog:
Useful for consolidated identity, networking, governance, procurement, and multiple providers. A catalog endpoint may expose a different feature set or release schedule from the model vendor’s native API.

Self-hosted open weights:
Best for control, data locality, tuning, and sustained-volume economics when utilization is high. The organization owns capacity planning, inference optimization, safety, upgrades, monitoring, and license compliance.

Edge/on-device:
Useful for privacy, offline use, and very low network latency. Model size, battery, memory, context, and update distribution are constraints.

Routing:
Use rules or a learned router based on risk, complexity, modality, language, context size, latency objective, and budget. Keep a safe fallback. Evaluate the router itself: wrong routing can erase savings or reduce quality. A common cascade is small utility model → balanced generator → reasoning/frontier escalation → human review.

title: Lifecycle and migration playbook
block_type: guide
provider: cross-provider
model_id: 
as_of_date: 2026-07-10
tags: llm-landscape, rag, guide, lifecycle-and-migration-playbook
source_ids: 
---CONTENT---
Model retirement is a normal operating condition.

Maintain an inventory of every model ID, alias, region, API version, prompt, tool schema, embedding index, evaluation result, and owner. Subscribe to provider lifecycle notices. Avoid undocumented aliases in production. Set internal support windows earlier than the vendor’s final date.

Migration steps:
1. Select a candidate replacement and confirm feature parity.
2. Run offline retrieval and generation regression tests.
3. Recalculate tokenizer usage and cost.
4. Test structured output, function calling, safety, multilingual behavior, context limits, and streaming.
5. Shadow production traffic without affecting users.
6. Canary a small percentage with rollback.
7. Compare human-rated quality and operational metrics.
8. Update prompts rather than assuming drop-in behavior.
9. Re-embed the corpus if the embedding model changes; do not mix incompatible vector spaces.
10. Archive the evaluation and decision record.

Use explicit dates in documentation. “Latest” and “current” are not reproducible identifiers.

Enterprise policy assistant:

Route intent with a fast model, retrieve current product and policy evidence, generate with a balanced model, and escalate sentiment, refund, medical, legal, or security issues. Measure resolution, correctness, and escalation—not only fluency.

Research assistant:
Separate private sources from public web search. Prefer primary sources, store retrieval date, validate citations, and use a reasoning or deep-research model only when the complexity justifies it.

High-stakes domain:
Restrict the system to decision support, retrieve only approved evidence, require calibrated uncertainty and human review, and conduct domain-specific safety validation.

Quantization: representing model weights at lower precision to reduce memory and accelerate inference.
RAG: retrieval-augmented generation.
Reranker: second-stage model that scores query-document relevance.
SLM: small language model.
TCO: total cost of ownership.
Tool calling: model-generated structured request to application code.
Vector database: system that indexes and searches embeddings.
